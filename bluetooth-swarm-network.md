# Bluetooth Swarm Network

This skill generates complete Bluetooth Low Energy mesh network applications using only vanilla JavaScript and vanilla HTML. Every output must be zero-dependency, fully offline, and contained within a 400KB total code budget. The skill covers disaster monitoring, human safety, livestock protection, surveillance-resistant communication, and distributed environmental sensing through swarm intelligence principles.

This skill exists because centralized infrastructure fails exactly when it is needed most. Cell towers collapse in earthquakes. Internet cables burn in wildfires. Cloud servers become surveillance tools under authoritarian regimes. The Bluetooth Swarm Network operates without any of these dependencies. It turns ordinary smartphones and cheap BLE devices into a self-healing mesh that can detect wildfires, relay SOS signals, monitor livestock, predict epidemics, warn of crowd crushes, and enable activists to communicate invisibly.

The fundamental insight is biological. Ant colonies have no central brain. Fungal networks have no server. Flocks of starlings have no leader. Yet these systems solve problems that would overwhelm any centralized architecture. This skill applies the same principles to BLE radio, creating emergent intelligence from simple local rules executed by many nodes.


SECTION 1: ARCHITECTURE OVERVIEW

1.1 Design Philosophy

Every line of code generated by this skill must satisfy three constraints simultaneously.

First, offline permanence. The system must function with zero internet connectivity indefinitely. Not as a fallback mode. Not as a degraded state. Offline is the primary and only expected operating environment.

Second, radical simplicity. Any NGO field worker with basic HTML knowledge must be able to read, modify, and deploy the code. This means no build tools, no transpilers, no package managers, no frameworks. Vanilla JavaScript and vanilla HTML only.

Third, ethical alignment. No data exploitation. No tracking beyond what the user explicitly consents to. No metadata leakage. All data must be deletable instantly. These are not optional features. They are architectural constraints that shape every design decision.

1.2 Network Topology

The swarm uses a managed flooding protocol with intelligent TTL limits. This is not a naive broadcast. Each message carries a unique identifier, a TTL counter that decrements at each hop, and a trust score derived from the sending node. Duplicate messages are dropped using a rolling bloom filter that consumes approximately 256 bytes of memory.

The topology is self-healing. When a node disappears, its neighbors detect the absence through missing heartbeat signals and automatically redistribute relay responsibilities. There is no single point of failure because there is no center.

Message propagation follows a probabilistic relay model. Each node does not blindly rebroadcast every message. Instead, it calculates a relay probability based on the number of neighbors that have already forwarded the same message. If many neighbors have already relayed, the probability drops. This prevents broadcast storms while maintaining coverage.

1.3 Node Types

There are four node types in the swarm.

Sentinel nodes carry sensors and generate data. A smartphone running the wildfire detection module is a Sentinel. A BLE-equipped livestock tag is a Sentinel. These nodes observe and report.

Relay nodes exist solely to extend range. A cheap BLE beacon placed on a tree or fencepost can serve as a Relay. It has no sensors. It receives messages and forwards them according to the relay probability algorithm. Relay nodes are disposable and replaceable.

Gateway nodes bridge the BLE mesh to external networks. A smartphone with cellular connectivity, a device connected to WiFi, or a LoRa transceiver can serve as a Gateway. Gateways translate swarm messages into SMS, HTTP, or other protocols. The swarm does not depend on Gateways to function internally, but Gateways enable the mesh to reach the outside world when connectivity exists.

Commander nodes are operated by humans who need to send instructions into the swarm. A field coordinator can use a Commander to trigger specific behaviors across the mesh, such as switching all nodes from CALM to ALERT mode, or requesting a roll call of active Sentinels.

Any physical device can serve multiple roles simultaneously. A smartphone can be a Sentinel, a Relay, and a Gateway at the same time.

1.4 The Heartbeat System

Every node in the swarm emits periodic BLE advertisements called heartbeats. The heartbeat is the fundamental unit of swarm communication. It serves three purposes: presence announcement, data transport, and network health monitoring.

Heartbeats operate in three modes.

CALM mode is the default. Heartbeat interval is 5 to 10 seconds. The advertisement payload contains only the node identifier, battery level, and a minimal status byte. Power consumption is approximately 0.5 milliwatts. A smartphone battery can sustain CALM mode for 48 to 72 hours with the screen off.

ALERT mode activates when anomaly detection thresholds are crossed. Heartbeat interval drops to 500 milliseconds to 1 second. The payload expands to include sensor readings, anomaly scores, and directional data. ALERT mode propagates through the mesh via cascade. When a node enters ALERT, its neighbors evaluate the alert data and may also transition to ALERT if the evidence supports it. This cascade behavior enables the swarm to collectively detect and track threats.

PANIC mode is the maximum urgency state. Heartbeat interval drops to 250 milliseconds. The payload includes full emergency data. PANIC mode triggers immediate relay to all Gateways with highest priority. PANIC is reserved for confirmed life-threatening situations such as active wildfire front, confirmed SOS from a human, or seismic event above threshold.

The transition rules between modes are deterministic. CALM to ALERT requires a local anomaly score exceeding the configured threshold. ALERT to PANIC requires either a local anomaly score exceeding the PANIC threshold or confirmation from at least three neighboring nodes also in ALERT mode for the same threat type. PANIC to ALERT requires the anomaly score dropping below the PANIC threshold for at least 60 consecutive seconds. ALERT to CALM requires the anomaly score dropping below the ALERT threshold for at least 300 consecutive seconds.

1.5 The Friendship System (Chinbun Trust Protocol)

Not all nodes in the swarm are equally trustworthy. A node that has been online for months, consistently relaying messages, and rarely producing false alerts is more trustworthy than a node that appeared five minutes ago.

The Friendship system assigns each node a trust score based on observed behavior. The score is calculated locally by each node based on its own interactions with neighbors. There is no central trust authority.

Trust score formula: trust = success_rate multiplied by time_decay multiplied by consistency_factor.

Success rate is the ratio of useful interactions (successful relays, confirmed alerts, valid data) to total interactions with that node. Time decay is an exponential decay function that gives more weight to recent behavior. A node that was reliable six months ago but erratic yesterday scores lower than a node with consistent recent performance. Consistency factor measures the variance in the node signal. Nodes that appear and disappear unpredictably score lower than nodes with stable presence patterns.

The trust score affects relay probability. Messages from high-trust nodes are relayed with higher probability. Messages from low-trust nodes may be relayed with lower probability or subjected to additional validation before relay.

This system prevents Sybil attacks where an adversary floods the network with fake nodes. New nodes start with a neutral trust score and must earn trust through consistent good behavior over time.

1.6 Platform Constraints

Web Bluetooth API is the primary interface. This imposes critical constraints that all generated code must respect.

iOS does not support Web Bluetooth. All code generated by this skill targets Android and desktop platforms. When generating user-facing applications, the code must include a clear platform check at startup and display an appropriate message for iOS users explaining the limitation.

Background BLE scanning is limited in browsers. The screen must remain active for continuous operation. Generated code must implement the Screen Wake Lock API to prevent the screen from turning off during active monitoring. For production deployments, the skill should note that a Capacitor or TWA wrapper can enable background BLE operation on Android.

BLE scanning requires a user gesture to initiate. The generated code must include a visible start button that the user taps to begin scanning. Automatic scanning on page load will fail silently in most browsers.

BLE advertisement payload is limited to 31 bytes in legacy mode and up to 255 bytes in extended advertisements. All data encoding must be extremely compact. The skill uses bitpacked binary formats, not JSON, for over-the-air data.

Maximum simultaneous BLE connections vary by device but are typically limited to 7 to 10. The swarm protocol must be designed to work primarily through connectionless advertisements rather than connected GATT sessions.

Module 04: Wildfire Monitor. Approximately 14KB. Multi-sensor fusion for fire detection including camera analysis, microphone crackling detection, ambient light spike detection, temperature monitoring, and barometric pressure analysis.

Module 05: Livestock Guardian. Approximately 15KB. Audio FFT for distress vocalization detection, RSSI-based movement tracking, heartbeat anomaly detection, and virtual fence geofencing.

Module 06: Human SOS Relay. Approximately 12KB. Covert activation through triple-tap or shake gesture, steganographic BLE encoding, multi-hop emergency relay to nearest Gateway.

Module 07: Disaster Detector. Approximately 14KB. Seismic pattern recognition from accelerometer, barometric pressure analysis for storms, audio signature detection for floods, and locust swarm detection.

Module 08: AI Engine Suite. Approximately 22KB. Contains six lightweight AI algorithms: Bayesian Anomaly Detector, Kalman Filter, FFT Audio Analyzer, Swarm Consensus Engine, Temporal Pattern Recognizer, and Friendship Trust Scorer.

Module 09: Acoustic Triangulation. Approximately 10KB. Uses time-difference-of-arrival across multiple nodes to locate the source of gunshots, explosions, or other sudden loud sounds.

Module 10: Steganographic Communication. Approximately 8KB. Hides human-readable messages inside normal-looking BLE advertisement data. Enables activists to communicate through the mesh without detectable message content.

2.1 Module Output Format

Every module generates code in the following format. This is critical. The output must not contain html, head, or body tags. It must be insertable into an existing index.html file as a block.

The format is:

Opening style tag.
All CSS for the module.
Closing style tag.

Opening div tag with a unique module identifier.
All HTML structure for the module.
Closing div tag.

Opening script tag.
All JavaScript for the module wrapped in an IIFE (Immediately Invoked Function Expression) to prevent global namespace pollution.

The div must use a consistent naming convention: swarm-module-[name]. For example, swarm-module-wildfire, swarm-module-sos, swarm-module-dashboard.

All CSS selectors must be scoped to the module div to prevent style conflicts with other modules or the host page.

2.2 SwarmBus Event System

Modules communicate through a global event bus. This is the only shared global object. Every module listens for events it cares about and emits events when it has data to share.

The SwarmBus implementation is approximately 40 lines of JavaScript. It provides three methods: on(eventName, callback), off(eventName, callback), and emit(eventName, data).

Core events that all modules should know about:

swarm:node-discovered. Emitted when a new BLE device is detected. Payload contains device identifier, RSSI, and raw advertisement data.

swarm:heartbeat-received. Emitted when a heartbeat from a known node is decoded. Payload contains node identifier, mode, battery, status, and any sensor data.

swarm:mode-change. Emitted when the local node transitions between CALM, ALERT, and PANIC. Payload contains previous mode, new mode, and trigger reason.

swarm:alert. Emitted when any module detects an anomaly worthy of alert. Payload contains alert type, severity, confidence score, and sensor data.

swarm:message-relay. Emitted when a message needs to be relayed through the mesh. Payload contains the complete message with TTL and routing data.

swarm:trust-update. Emitted when a node trust score changes significantly. Payload contains node identifier, old score, new score, and reason.

swarm:gateway-available. Emitted when a Gateway connection (cellular, WiFi) becomes available. Modules can queue outbound messages for Gateway transmission.

2.3 Data Storage

All persistent data uses IndexedDB through a thin wrapper. The wrapper provides simple get, set, delete, and query operations. No external database library is used.

The IndexedDB schema has the following object stores:

nodes. Stores known node records including identifier, last seen timestamp, trust score, node type, and accumulated statistics.

alerts. Stores alert history including timestamp, type, severity, source node, sensor data, and resolution status.

messages. Stores message queue for relay including message content, TTL, origin, destination, and relay count.

config. Stores user configuration including alert thresholds, mode preferences, node role assignment, and sensor calibration data.

sensor_log. Stores rolling sensor data for temporal pattern analysis. This store uses a circular buffer pattern, automatically deleting records older than a configurable retention period (default 24 hours).

Total IndexedDB usage should not exceed 50MB to remain viable on low-end devices.

2.4 Deletion Protocol

Aligned with Principle 10 (Collect no data and allow simple deletion), every module must implement a complete data deletion function. A single function call must erase all IndexedDB stores, all localStorage entries, all sessionStorage entries, and all in-memory data structures. After deletion, the application state must be indistinguishable from a fresh installation.

The deletion function must be accessible from the UI dashboard with a clearly labeled button. No confirmation dialog beyond a single tap. Speed of deletion is a security feature.


SECTION 3: CORE MESH ENGINE (MODULE 01)

3.1 BLE Scanning Implementation

The core mesh engine initializes the Web Bluetooth scanning subsystem. On user gesture (button tap), it calls navigator.bluetooth.requestLEScan with appropriate filters.

The scan filter should accept advertisements containing a specific service UUID that identifies swarm participants. Use a custom 128-bit UUID to avoid collision with standard BLE services. The recommended UUID format is: prefix-SWRM-suffix where the middle portion identifies this as a swarm network participant.

When an advertisement is received, the engine performs the following steps in order.

Step 1: Extract the raw manufacturer data or service data from the advertisement.

Step 2: Check the rolling bloom filter for the message unique identifier. If the identifier exists in the bloom filter, this is a duplicate. Drop it silently.

Step 3: If not a duplicate, add the identifier to the bloom filter.

Step 4: Decode the advertisement payload according to the heartbeat protocol format.

Step 5: Update the node record in the nodes IndexedDB store.

Step 6: Emit the appropriate SwarmBus event (swarm:node-discovered for new nodes, swarm:heartbeat-received for known nodes).

Step 7: Evaluate whether this message should be relayed. Calculate relay probability based on: TTL remaining (if zero, do not relay), number of neighbors that have already relayed this message (estimated from bloom filter collision rate), and trust score of the originating node.

Step 8: If relay decision is positive, decrement TTL and schedule retransmission after a random jitter delay between 50 and 500 milliseconds. The jitter prevents simultaneous retransmission from multiple nodes.

3.2 BLE Advertising Implementation

To advertise, the engine uses the Web Bluetooth API advertising capabilities where available. On platforms where advertising is not directly supported through the Web API, the engine falls back to a GATT server mode where it creates a BLE peripheral with a readable characteristic containing the heartbeat data.

The advertising payload is packed into a maximum of 31 bytes for legacy compatibility. The byte layout is:

Bytes 0-1: Swarm protocol identifier (2 bytes, fixed magic number 0x5357 for SW representing Swarm).
Byte 2: Protocol version (1 byte, current version 0x01).
Bytes 3-6: Node identifier (4 bytes, generated once and stored in localStorage).
Byte 7: Node type and mode (upper 2 bits = node type, lower 2 bits = heartbeat mode, remaining 4 bits = flags).
Byte 8: Battery level (1 byte, 0-100 percentage).
Byte 9: Trust score summary (1 byte, 0-255 mapped from 0.0-1.0).
Bytes 10-11: Message sequence number (2 bytes, wrapping counter).
Byte 12: TTL (1 byte, default starting value 7).
Bytes 13-14: Alert type and severity (if in ALERT or PANIC mode).
Bytes 15-30: Payload data (variable, depends on mode and alert type).
Byte 31: CRC-8 checksum of bytes 0-30.

This packing fits within the 31-byte legacy advertisement limit while carrying substantial information.

3.3 Bloom Filter for Duplicate Detection

The bloom filter uses 256 bytes (2048 bits) with 3 hash functions. This provides a false positive rate of approximately 1% for up to 500 recent messages, which is sufficient for most swarm densities.

The hash functions use simple bitwise operations on the message identifier bytes. No cryptographic hash is needed for duplicate detection.

The bloom filter is cleared every 60 seconds to prevent saturation. This means a message that was seen more than 60 seconds ago may be accepted again, but this is acceptable because legitimate messages have TTL limits that prevent infinite circulation.

3.4 Managed Flooding Algorithm

The flooding algorithm determines whether to relay a received message. The decision is probabilistic, not deterministic.

Base relay probability starts at 0.8 (80%).

Adjustment factors:

TTL factor: If TTL is 1 (last hop), probability increases to 0.95 regardless of other factors. This ensures critical messages reach maximum range.

Density factor: Count the number of unique nodes heard in the last 30 seconds. If this count exceeds 10, reduce probability by 0.05 for each additional node above 10, to a minimum of 0.3. Dense areas need less redundant relay.

Trust factor: Multiply probability by the originating node trust score. Messages from nodes with trust 0.9 get nearly full relay probability. Messages from nodes with trust 0.3 get significantly reduced probability.

Battery factor: If local battery is below 20%, reduce probability by 0.3. If below 10%, reduce by 0.5. A dying node should conserve energy rather than relay.

PANIC override: If the message mode is PANIC, relay probability is always 1.0 regardless of all other factors. PANIC messages are always relayed.

3.5 Code Structure for Core Mesh Engine

When generating the Core Mesh Engine module, the code must follow this structure:

The IIFE wraps all code. At the top, declare constants for the magic number, protocol version, UUID, bloom filter size, and default configuration values.

Next, implement the BloomFilter class with add, check, and clear methods.

Next, implement the SwarmBus singleton if it does not already exist on the window object.

Next, implement the MeshEngine class with methods: startScan, stopScan, startAdvertising, stopAdvertising, handleAdvertisement, shouldRelay, relayMessage, encodePayload, decodePayload, updateNodeRecord, and getNetworkStats.

Next, implement the IndexedDB wrapper for the nodes and messages stores.

Finally, create the UI elements: a start/stop scan button, a network status indicator showing node count and mode, and a minimal log area showing recent events.

The CSS should style these elements as a compact card that fits within a dashboard layout. Use a dark theme with green accent colors to suggest a monitoring system aesthetic.


SECTION 4: HEARTBEAT PROTOCOL (MODULE 02)

4.1 Mode State Machine

The heartbeat protocol manages the three operating modes as a finite state machine. Each mode has an entry condition, an exit condition, a heartbeat interval, and a payload format.

State transitions are logged to IndexedDB for post-event analysis. Each log entry records: timestamp, previous mode, new mode, trigger reason (which sensor or algorithm triggered the transition), confidence score of the trigger, and the sensor reading that caused the trigger.

4.2 CALM Mode Payload

In CALM mode, the heartbeat payload (bytes 15-30 of the advertisement) contains:

Bytes 15-16: Ambient temperature (signed 16-bit integer, value multiplied by 10 for one decimal place precision, in Celsius).
Bytes 17-18: Barometric pressure (unsigned 16-bit integer, value in hectopascals minus 900, giving range 900-1155 hPa).
Byte 19: Ambient light level (unsigned 8-bit, logarithmic scale 0-255).
Byte 20: Noise level (unsigned 8-bit, decibel estimate 0-120 mapped to 0-255).
Bytes 21-24: GPS latitude if available (32-bit float, or 0xFFFFFFFF if no GPS).
Bytes 25-28: GPS longitude if available (32-bit float, or 0xFFFFFFFF if no GPS).
Bytes 29-30: Reserved for future use.

4.3 ALERT Mode Payload

In ALERT mode, the payload shifts to prioritize the alert data:

Byte 13: Alert type code. Defined values: 0x01 = fire, 0x02 = seismic, 0x03 = livestock distress, 0x04 = human SOS, 0x05 = weather severe, 0x06 = gunshot detected, 0x07 = crowd crush risk, 0x08 = air quality hazard, 0x09 = epidemic signal, 0x0A = fence breach, 0x0B = flood, 0x0C = locust swarm.
Byte 14: Confidence score (0-255 mapped to 0.0-1.0).
Bytes 15-18: Primary sensor reading (32-bit float, meaning depends on alert type).
Bytes 19-22: Secondary sensor reading (32-bit float).
Bytes 23-26: Timestamp offset from last midnight in seconds (allows 24-hour range in 4 bytes).
Bytes 27-28: Originating node identifier suffix (last 2 bytes for quick identification without full ID).
Bytes 29-30: Hop count and relay path hash (for routing analysis).

4.4 PANIC Mode Payload

PANIC mode uses the same format as ALERT mode but with the following differences:

The heartbeat interval is 250 milliseconds instead of 500 milliseconds to 1 second.

The TTL starting value increases from the default 7 to 15, ensuring PANIC messages reach across the entire mesh.

A special flag in byte 7 indicates PANIC mode, which causes all receiving nodes to immediately evaluate whether they should also enter PANIC mode based on the received sensor data and their own local readings.

4.5 Adaptive Heartbeat Interval

Within each mode, the heartbeat interval is not fixed but adapts to conditions.

In CALM mode, if the node detects no other nodes nearby (isolated node), the interval extends to 30 seconds to conserve battery. If many nodes are nearby (dense area), the interval may extend to 15 seconds since redundant heartbeats from many nodes provide adequate coverage.

In ALERT mode, the interval tightens as confidence increases. At confidence 0.5, interval is 1 second. At confidence 0.8, interval is 500 milliseconds. At confidence 1.0, interval is 250 milliseconds (effectively PANIC rate).

This adaptive behavior emerges from a simple formula: interval = base_interval divided by (1 plus confidence multiplied by scaling_factor).


SECTION 5: FRIENDSHIP TRUST ENGINE (MODULE 03)

5.1 Trust Score Calculation

Each node maintains a local trust table for every node it has interacted with. The trust table is stored in IndexedDB and updated after every interaction.

An interaction is defined as any event where the local node receives data from another node and can evaluate its quality. Examples include: receiving a heartbeat (was it well-formed?), receiving a relayed alert (was it confirmed by other sources?), receiving a relay of our own message (did the node actually help propagate it?).

The trust score is a floating-point value between 0.0 and 1.0, calculated as follows:

Raw success rate = successful interactions divided by total interactions.

Time-weighted success rate = sum of (interaction_success multiplied by decay_factor raised to the power of age_in_hours) divided by sum of (decay_factor raised to the power of age_in_hours). The decay factor is 0.995 per hour, meaning interactions from 24 hours ago retain approximately 88% of their weight, while interactions from 7 days ago retain approximately 41%.

Consistency factor = 1.0 minus (standard deviation of recent interaction success values divided by maximum possible standard deviation). A node that alternates between perfect and terrible performance has lower consistency than one that is reliably mediocre.

Final trust score = time_weighted_success_rate multiplied by consistency_factor.

New nodes start with a trust score of 0.5 (neutral). This means their messages are relayed with moderate probability. They must demonstrate consistent good behavior to earn higher trust.

5.2 Trust Events

The following events increase trust: successful message relay confirmed by downstream acknowledgment, alert that is corroborated by independent sensor readings from other nodes, consistent heartbeat timing that matches expected intervals, battery level reporting that follows realistic discharge curves.

The following events decrease trust: malformed or unparseable advertisements, alerts that are contradicted by all neighboring nodes, heartbeat timing that is wildly inconsistent, advertisement data that changes node identifier (possible spoofing), excessive message generation that could indicate flooding attack.

5.3 Sybil Attack Resistance

The trust system provides natural Sybil resistance because trust must be earned over time. An attacker who creates many fake nodes cannot immediately influence the network because all fake nodes start at 0.5 trust and face reduced relay probability.

Additionally, the Friendship system implements a neighbor consistency check. If a previously unseen node suddenly appears with a high trust claim (possible in scenarios where trust data is shared), the local node ignores the claimed trust and assigns its own initial 0.5 score.

The system also detects impossible presence patterns. If a node claims to be in two distant locations simultaneously (based on RSSI readings from different parts of the mesh), its trust is immediately reduced to 0.1.


SECTION 6: WILDFIRE MONITOR (MODULE 04)

6.1 Detection Philosophy

Wildfire detection in this system does not rely on any single sensor. A camera can see orange light but also sees sunsets. A microphone can hear crackling but also hears campfires. A temperature sensor can detect heat but also detects hot pavement. No single sensor is reliable.

The wildfire monitor uses multi-sensor fusion to combine weak signals into strong evidence. The Bayesian Anomaly Detector (from the AI Engine Suite) maintains a running probability estimate for the hypothesis "wildfire is present" and updates it with each new sensor reading.

6.2 Sensor Channels

Camera channel: The module captures periodic frames from the device camera (if permissions are granted) and analyzes the color histogram. Wildfire produces a characteristic shift toward orange and red wavelengths. The analysis computes the ratio of pixels in the red-orange range (hue 0-30 in HSL) to total pixels. A ratio above 0.4 in an outdoor environment is a moderate fire indicator. A ratio above 0.7 is a strong indicator.

The camera analysis uses Canvas API to extract pixel data. Each frame is downsampled to 64x64 pixels before analysis to minimize CPU usage. Analysis frequency is once every 10 seconds in CALM mode, once per second in ALERT mode.

Microphone channel: The module uses the Web Audio API to capture ambient sound and perform FFT analysis. Wildfire produces a broadband crackling sound with energy concentrated in the 1000-4000 Hz range. The analysis computes the ratio of spectral energy in this range to total spectral energy. A ratio above 0.3 sustained for more than 5 seconds is a moderate indicator.

The FFT uses a 2048-point window, providing approximately 23 Hz resolution at 48000 Hz sample rate. An AudioWorklet processes the data in a separate thread to avoid blocking the main UI thread.

Temperature channel: The module reads the device temperature sensor if available through the Generic Sensor API. Most smartphones do not expose an ambient temperature sensor, but some devices report battery temperature which rises faster than ambient in fire proximity. A rate of temperature increase exceeding 2 degrees Celsius per minute is a moderate indicator.

Light channel: The Ambient Light Sensor API provides ambient light readings. A sudden spike in light level (more than 500% increase within 10 seconds) followed by sustained high levels is a moderate fire indicator, especially if correlated with the camera channel.

Barometric pressure channel: Fires create local low-pressure zones due to intense heat causing air to rise rapidly. A pressure drop exceeding 3 hPa within 5 minutes, if not correlated with normal weather patterns, is a weak but contributing indicator.

6.3 Multi-Sensor Fusion

The Bayesian update rule combines all sensor channels. Each channel provides a likelihood ratio: how much more likely is this reading under the wildfire hypothesis versus the no-wildfire hypothesis.

The prior probability starts at 0.001 (0.1%) in the absence of any alerts from neighboring nodes. If neighboring nodes have entered ALERT mode for fire, the prior increases to 0.1 (10%).

Each sensor channel updates the probability: posterior = (prior multiplied by likelihood) divided by ((prior multiplied by likelihood) plus ((1 minus prior) multiplied by (1 minus likelihood))).

The channels are processed sequentially. After all channels are processed, the final posterior probability determines the action:

Below 0.3: No action. Remain in CALM mode.
0.3 to 0.6: Log as possible anomaly. Increase sensor sampling frequency. Do not change mode.
0.6 to 0.8: Transition to ALERT mode. Begin cascade notification to neighbors.
Above 0.8: Transition to PANIC mode. Immediate relay to all Gateways.

6.4 Node Dropout Cascade Detection

This is unique to the swarm approach and is arguably more reliable than any single-node sensor.

The principle: when a wildfire burns through an area, the BLE nodes in that area stop transmitting. This creates a pattern of node dropout that is distinguishable from random failures.

Random failure: Nodes drop out independently, at random times, with no spatial correlation. Expect 1-2 nodes per day in a 100-node network due to battery death, physical damage, or wildlife interference.

Wildfire pattern: Multiple nodes drop out in rapid succession (within seconds to minutes), in a spatially correlated pattern. The dropout front advances in a direction consistent with wind patterns.

The wildfire monitor tracks node dropout events and applies a spatial-temporal correlation analysis. If 3 or more nodes within estimated mutual BLE range (approximately 50 meters) drop out within a 2-minute window, and the dropout pattern extends in a linear or arc-shaped front, the system interprets this as a fire front advancing through the node field.

This detection method works even if all sensor-based detection fails. Even if cameras are facing the wrong direction, even if microphones are too far from the fire to hear crackling, the death of the nodes themselves reveals the fire.

6.5 Wind Direction Estimation

The direction of fire spread can be estimated from the temporal order of node dropout. If Node A drops out first, then Node B (located 50 meters east of A) drops out 30 seconds later, then Node C (100 meters east of A) drops out 60 seconds later, the fire is moving eastward at approximately 1.5 meters per second.

This information is encoded in ALERT and PANIC payloads and relayed to Gateways, enabling fire response teams to know not just that a fire exists but where it is heading and how fast.


SECTION 7: LIVESTOCK GUARDIAN (MODULE 05)

7.1 Audio Distress Detection

Different livestock species produce characteristic vocalizations when distressed. The system performs real-time FFT analysis on audio captured from BLE-equipped devices near livestock.

Cattle distress calls are typically 100-400 Hz with higher amplitude and longer duration than normal vocalizations. The distress indicator is a sustained vocalization (more than 3 seconds) in this frequency range at amplitude 20dB or more above the baseline ambient noise level.

Sheep distress calls are typically 300-600 Hz. Rapid repeated bleating (more than 5 calls within 10 seconds) indicates predator presence or panic.

Goat distress is similar to sheep but often higher pitched, 400-800 Hz.

The audio analysis runs on an AudioWorklet to avoid blocking the main thread. The worklet performs a 1024-point FFT every 100 milliseconds and emits summary data (peak frequency, peak amplitude, spectral centroid, duration of sustained vocalization) to the main thread.

7.2 RSSI Movement Tracking

BLE signal strength (RSSI) varies with distance. While RSSI is notoriously noisy and imprecise for absolute distance measurement, it is useful for detecting movement patterns.

The system uses a Kalman Filter (from the AI Engine Suite) to smooth RSSI readings and estimate the relative distance trend for each animal. The Kalman Filter state includes estimated distance and estimated velocity (rate of distance change).

Normal movement: Slow, random variation in RSSI. The Kalman Filter velocity estimate stays near zero with occasional small deviations.

Stampede detection: Rapid, sustained decrease in RSSI from multiple animals simultaneously (they are all moving away). This triggers ALERT mode.

Predator circling: A pattern where RSSI from multiple animals shows simultaneous oscillation (animals clustering together) while one or two non-livestock BLE signals show rapid RSSI changes (predator moving around the herd). This is a sophisticated detection that requires the trust system to distinguish known livestock tags from unknown BLE devices.

7.3 Virtual Fence

The livestock guardian implements a virtual fence using BLE beacon boundaries. The farmer places BLE beacons at the boundary of the desired grazing area. The system monitors whether livestock tags are receiving signals from the boundary beacons.

When a livestock tag stops receiving signal from all boundary beacons (meaning it has moved beyond BLE range of all of them), a fence breach alert is generated.

The fence boundary is defined as a set of beacon identifiers in the configuration. The farmer registers boundary beacons through the UI dashboard by placing them and pressing a "register boundary" button.

The breach detection includes hysteresis to avoid false alarms: the livestock must be out of range of all boundary beacons for at least 60 consecutive seconds before a breach alert is generated. Brief signal losses due to obstructions are ignored.

7.4 Health Anomaly Detection

If livestock carry BLE sensors capable of transmitting heart rate (some commercial livestock tags support this), the system monitors for anomalies in heart rate patterns.

The Temporal Pattern Recognizer (from the AI Engine Suite) maintains a baseline heart rate profile for each animal across the 24-hour cycle. Significant deviation from the baseline (more than 2 standard deviations sustained for more than 5 minutes) triggers a health alert.

Heart rate alone is insufficient for diagnosis, but sustained tachycardia can indicate fever, pain, or extreme stress. Combined with movement data (animal is stationary despite elevated heart rate), this suggests illness rather than exertion.


SECTION 8: HUMAN SOS RELAY (MODULE 06)

8.1 Covert Activation

The SOS system is designed for situations where the user cannot openly use their phone. Possible scenarios include: a person trapped under earthquake rubble, an activist being detained, a kidnapping victim, a worker in a dangerous environment.

The system supports three covert activation methods.

Triple-tap: Three rapid taps on the phone (detected by accelerometer) within 1.5 seconds. The taps must have peak acceleration above 2g each. This can be performed with the phone in a pocket by tapping on the outside of the pocket.

Shake pattern: A specific shake sequence (for example, two short shakes followed by one long shake, like Morse code for "SOS") detected by accelerometer. The pattern must match within a tolerance of 500 milliseconds per element.

Volume button sequence: A specific sequence of volume-up and volume-down presses (for example, up-up-down-down-up). This is detected by listening for volumechange events on a media element.

All three methods are configurable. The user sets their preferred activation method in the configuration. Only the configured method is active to prevent accidental activation.

Upon activation, the phone provides minimal feedback: a single brief vibration (50 milliseconds) to confirm activation. No visible change on screen. No sound. If the phone is face-down, even the vibration can be suppressed and replaced with a barely perceptible screen flash.

8.2 SOS Payload

The SOS message contains: node identifier (so rescuers can identify who is sending), GPS coordinates if available, timestamp, battery level, and a 4-bit severity code (0000 = test, 0001 = need help, 0010 = medical emergency, 0011 = immediate danger to life, and further codes reserved).

The entire SOS payload fits within the standard 31-byte advertisement format with the alert type code set to 0x04 (human SOS).

8.3 Multi-Hop Emergency Relay

SOS messages receive the highest relay priority. All nodes in the mesh, regardless of battery level, trust calculations, or density factors, relay SOS messages with probability 1.0. The TTL for SOS messages is set to the maximum value of 15.

The relay continues until the message reaches a Gateway node. The Gateway transmits the SOS via any available external channel: SMS to a preconfigured emergency number, HTTP POST to a rescue coordination server, or simply stores it for retrieval by search teams.

If no Gateway is available, the SOS message continues circulating in the mesh. Any device that later gains connectivity will transmit all buffered SOS messages.

8.4 Rescue Coordination

Once an SOS is received at a Gateway, the system can provide navigation assistance to rescue teams. The mesh knows (approximately) the location of the SOS sender based on RSSI triangulation across nodes that received the original SOS.

The Gateway generates a rescue beacon: a BLE advertisement containing the estimated coordinates of the person in distress. Rescue team members with the dashboard app can follow the signal strength toward the sender, using the mesh itself as a localization system.

8.5 Steganographic SOS Encoding

For situations where even the existence of an SOS system must be concealed (activist detention, etc.), the SOS can be encoded steganographically into what appears to be normal BLE traffic. See Module 10 (Steganographic Communication) for the full encoding protocol. The SOS module interfaces with the steganography module when the user configuration enables "concealed mode."


SECTION 9: DISASTER DETECTOR (MODULE 07)

9.1 Seismic Detection

Smartphones contain accelerometers capable of detecting seismic waves. The challenge is distinguishing earthquake vibrations from normal phone movement (walking, driving, being placed on a table).

The system uses a two-stage approach.

Stage 1: The accelerometer is sampled at the maximum available rate (typically 60-100 Hz on modern smartphones). A high-pass filter removes DC offset and slow drift. The remaining signal is analyzed for seismic signatures.

Stage 2: Seismic waves have characteristic frequency content. P-waves (primary, compressional) typically range from 1-10 Hz. S-waves (secondary, shear) typically range from 0.5-5 Hz. The system applies bandpass filtering to isolate energy in these ranges.

The key discriminator is the multi-node correlation. A single phone vibrating could be anything. But if 10 phones within BLE range all detect the same vibration pattern at the same time, it is almost certainly seismic. The swarm consensus engine (from AI Engine Suite) collects accelerometer anomaly reports from neighboring nodes and votes on whether a seismic event is occurring.

The threshold for single-node seismic alert is a sustained vibration (more than 3 seconds) in the seismic frequency bands with amplitude exceeding 0.05g, while the phone is stationary (no GPS velocity, no user interaction for at least 30 seconds prior).

The threshold for swarm-confirmed seismic alert is 3 or more nodes within BLE range reporting seismic signatures within a 5-second window.

9.2 Barometric Storm Detection

Rapid drops in barometric pressure indicate approaching severe weather. The system reads the barometric pressure sensor (available on many Android devices through the Generic Sensor API).

Normal pressure change: less than 1 hPa per hour.
Moderate weather approaching: 1-3 hPa per hour.
Severe weather: more than 3 hPa per hour.
Extreme event (hurricane, typhoon): more than 5 hPa per hour.

The system maintains a 6-hour rolling average and alerts when the current reading deviates significantly from the rolling average adjusted for normal diurnal variation.

9.3 Flood Detection

Floods produce characteristic low-frequency sound. Rushing water generates broadband noise with significant energy below 500 Hz. The audio analysis module can detect this signature.

Additionally, if GPS is available, a sudden change in altitude reading (more than 2 meters in 5 minutes) while the phone is stationary may indicate rising water levels around the device.

The swarm-level flood detection combines audio signatures across multiple nodes. If nodes along a river valley sequentially detect rushing water sounds (downstream nodes first, upstream nodes later), this indicates a flood wave propagating upstream or a dam break.

9.4 Locust Swarm Detection

Locust swarms produce a characteristic buzzing sound in the 50-200 Hz range. The FFT audio analysis can detect this frequency signature.

The swarm-level confirmation is critical here because many common sounds (engines, wind, distant traffic) also produce energy in this range. The discriminator is spatial extent: if 5 or more nodes spread across at least 200 meters all detect the same 50-200 Hz signature simultaneously, this is consistent with a locust swarm but not with any single localized noise source.

The alert payload for locust detection includes the estimated directional heading of the swarm (determined by which nodes detect the sound first versus last) and the estimated width of the swarm front (determined by the spread of detecting nodes perpendicular to the heading).


SECTION 10: AI ENGINE SUITE (MODULE 08)

10.1 Bayesian Anomaly Detector

Size: approximately 2KB of JavaScript. Storage: approximately 100 bytes per tracked hypothesis in localStorage.

The Bayesian Anomaly Detector maintains probability estimates for hypotheses (fire present, earthquake occurring, flood approaching, etc.) and updates them using Bayes theorem as new sensor data arrives.

Implementation: A JavaScript class with the following methods.

Constructor takes an initial prior probability (default 0.001 for rare events).

Update method takes a likelihood ratio (how much more likely is this observation if the hypothesis is true versus false) and applies the Bayesian update formula: posterior equals (prior times likelihood) divided by ((prior times likelihood) plus ((1 minus prior) times (1 minus likelihood))).

The likelihood ratios for each sensor channel are preconfigured based on empirical data. For example, seeing an orange camera histogram is 15 times more likely during a fire than during normal conditions (likelihood ratio 15). Hearing crackling in the 1000-4000 Hz range is 8 times more likely during a fire (likelihood ratio 8).

Reset method returns the probability to the initial prior.

Serialize and deserialize methods allow the state to be saved to and restored from localStorage.

10.2 Kalman Filter

Size: approximately 1.5KB of JavaScript. Approximately 30-50 lines of core logic.

The Kalman Filter is used for RSSI tracking (estimating animal distance and velocity from noisy RSSI readings) and for sensor fusion (combining multiple noisy sensor readings into a single best estimate).

Implementation: A JavaScript class implementing the standard 1D Kalman Filter with the following state: estimated value (x), estimated uncertainty (P), process noise (Q), measurement noise (R).

The predict method advances the state by one time step: x stays the same (assume constant value model), P increases by Q.

The update method incorporates a new measurement: compute Kalman gain K = P / (P + R), update x = x + K * (measurement - x), update P = (1 - K) * P.

For the RSSI tracking application, the state is extended to 2D: position and velocity. The predict step applies the velocity to advance the position estimate. The update step incorporates the new RSSI reading converted to an approximate distance.

10.3 FFT Audio Analyzer

Size: approximately 3KB of JavaScript in the AudioWorklet, approximately 2KB of JavaScript in the main thread for setup and result processing.

The FFT is implemented using the Web Audio API AnalyserNode, which provides built-in FFT computation. The skill does not implement FFT from scratch because the Web Audio API provides a highly optimized native implementation.

Implementation: Create an AudioContext, connect the microphone input to an AnalyserNode with fftSize set to 2048 (providing 1024 frequency bins). Call getFloatFrequencyData at regular intervals to retrieve the frequency spectrum.

For real-time processing, an AudioWorklet is used. The worklet receives raw audio samples, computes additional metrics (zero-crossing rate, spectral centroid, spectral rolloff), and posts summary data to the main thread.

The main thread receives the summary data and routes it to the appropriate detection module (wildfire crackling detector, livestock vocalization detector, gunshot detector, flood detector, locust detector).

10.4 Swarm Consensus Engine

Size: approximately 2KB of JavaScript.

The consensus engine implements a simple voting protocol. When a node detects an anomaly, it broadcasts an alert vote. Neighboring nodes that also detect the anomaly broadcast their own votes. Each node tallies the votes from its neighbors.

The consensus threshold is configurable, default is: at least 3 nodes voting for the same alert type within a 30-second window constitutes consensus.

Consensus elevates the alert level. A single-node alert might trigger ALERT mode locally. Consensus-confirmed alert triggers cascade ALERT across all participating nodes. Consensus at the PANIC level (many nodes in ALERT with high confidence) triggers PANIC mode.

The voting protocol includes a vote expiry. Votes older than 60 seconds are discarded. This prevents stale votes from influencing current consensus.

10.5 Temporal Pattern Recognizer

Size: approximately 3KB of JavaScript.

The temporal pattern recognizer maintains a circular buffer of sensor readings and detects patterns over time. The buffer holds 100 readings per sensor channel, with configurable sampling intervals.

The pattern recognition algorithm is simple but effective: it computes a moving average and a moving standard deviation. When the current reading exceeds the moving average by more than a configurable number of standard deviations, it is flagged as an anomaly.

The recognizer also detects trends: if the moving average itself is increasing or decreasing steadily over time, this indicates a gradual change that may warrant attention even if no single reading is anomalous.

For diurnal cycle compensation, the recognizer maintains separate baselines for each hour of the day (if sufficient historical data is available). This prevents regular daily patterns (sunrise temperature increase, evening noise decrease) from triggering false alarms.

10.6 Friendship Trust Scorer

Size: approximately 1.5KB of JavaScript.

This is the computational core of the Friendship Trust Engine (Module 03). It implements the trust score formula described in Section 5 as a pure function that takes a node interaction history and returns a trust score.

The interaction history is stored as an array of records: timestamp, interaction type, outcome (success or failure), and metadata.

The scorer computes time-weighted success rate, consistency factor, and final trust score as described in Section 5.1.


SECTION 11: ACOUSTIC TRIANGULATION (MODULE 09)

11.1 Time-Difference-of-Arrival Principle

When a loud sound (gunshot, explosion, tree falling) occurs, it reaches different nodes at slightly different times based on their distance from the source. By measuring these time differences, the source location can be estimated.

The speed of sound is approximately 343 meters per second at sea level and 20 degrees Celsius. A time difference of 1 millisecond corresponds to approximately 34 centimeters of distance difference.

For a useful triangulation, at least 3 nodes must detect the same sound event, and their positions must be known (either through GPS or through manual placement data in the configuration).

11.2 Sound Event Detection

Each node continuously monitors the audio stream for sudden loud events. The detection threshold is a spike in audio amplitude that exceeds the ambient noise level by at least 30dB and rises from baseline to peak within 50 milliseconds.

When such an event is detected, the node records: the precise timestamp (using performance.now() for sub-millisecond precision), the peak amplitude, the spectral signature of the event (FFT snapshot at the moment of peak), and the timestamp is transmitted in the heartbeat payload.

11.3 Triangulation Calculation

When a Gateway or Commander node receives event reports from 3 or more nodes for what appears to be the same event (similar spectral signature within a time window consistent with the maximum possible propagation delay across the mesh), it performs triangulation.

The triangulation uses the TDOA (Time Difference of Arrival) algorithm. Given 3 nodes at known positions (x1,y1), (x2,y2), (x3,y3) and time differences dt12, dt13:

The distance differences are: d12 = speed_of_sound * dt12, d13 = speed_of_sound * dt13.

This defines two hyperbolas. The intersection of the hyperbolas is the estimated source location. The calculation reduces to solving a system of two equations which can be solved analytically.

11.4 Gunshot vs Other Sounds Classification

Gunshots have a distinctive acoustic signature: a very sharp attack (rise time less than 1 millisecond), high peak amplitude, and a characteristic spectral shape with energy concentrated in the 500-5000 Hz range and a rapid decay.

Explosions have a longer attack time (5-20 milliseconds), lower frequency emphasis (below 500 Hz), and longer sustained energy.

Thunder has a very long attack time (50-500 milliseconds) and extremely low frequency emphasis (below 200 Hz).

Tree falling has moderate attack time (10-100 milliseconds), broadband spectrum, and extended duration (1-5 seconds).

The classifier uses a simple decision tree based on: attack time, peak frequency, spectral centroid, and total event duration. This requires no machine learning training and fits in approximately 50 lines of JavaScript.

11.5 Accuracy Limitations

The practical accuracy of acoustic triangulation in this system depends on several factors.

Time synchronization: BLE nodes do not share a precise common clock. The timestamps are relative to each node's local clock, which may drift. To mitigate this, the system periodically synchronizes clocks by having nodes broadcast a time reference that neighbors can use to estimate clock offset.

Node position accuracy: GPS accuracy is typically 5-10 meters. This limits triangulation accuracy to approximately 10-20 meters in ideal conditions.

Multipath reflections: In built-up areas or forests, sound reflects off surfaces, creating delayed arrivals that confuse the triangulation. The system mitigates this by using the first arrival (direct path) rather than the strongest signal (which may be a reflection).

Practical accuracy: In open terrain with good GPS, the system can estimate the source location within 20-50 meters. In forested or urban environments, accuracy degrades to 50-200 meters. This is sufficient to direct responders to the correct area.


SECTION 12: STEGANOGRAPHIC COMMUNICATION (MODULE 10)

12.1 Purpose

In surveillance environments, even the existence of a communication channel can endanger activists. Standard encrypted messaging protects content but not the fact that communication is occurring. Steganographic communication hides the message inside normal-looking traffic so that observers cannot determine whether communication is taking place at all.

12.2 BLE Advertisement Steganography

BLE advertisements contain several fields that are expected to vary normally: RSSI (varies with environment), manufacturer-specific data (varies by device), and timing (varies with battery and implementation).

The steganographic encoder embeds message bits in the least significant bits of these normally varying fields. An observer monitoring BLE traffic would see advertisements that look completely normal. The variations that carry hidden data are indistinguishable from natural variation.

Encoding scheme: Each heartbeat advertisement carries 4 hidden bits in the following locations.

Bit 1: The least significant bit of the battery level byte. Battery readings naturally fluctuate by plus or minus 1%, so a single LSB change is invisible.

Bit 2: The parity of the heartbeat sequence number. Odd or even sequence numbers are equally normal, so controlling parity conveys one bit.

Bit 3: The timing offset of the advertisement from the expected interval. The encoder delays the advertisement by 0 or 1 millisecond. This tiny variation is well within normal BLE timing jitter.

Bit 4: The least significant bit of the noise level reading. Noise readings naturally fluctuate.

At the CALM mode heartbeat rate of one advertisement every 5-10 seconds, this yields 4 bits per 5-10 seconds. Transmitting a 140-character message (approximately 1120 bits at 8 bits per character) would take approximately 1400 to 2800 seconds (23 to 47 minutes).

This is slow by design. Steganographic communication prioritizes undetectability over speed. For urgent messages, the SOS system (Module 06) should be used instead.

12.3 Message Framing

Hidden bits are assembled into a message using a simple framing protocol.

Frame start: 8 consecutive 1-bits (an unlikely natural sequence, serving as synchronization).
Frame length: 16 bits indicating the number of payload bytes.
Payload: The message bytes.
Frame end: 8 consecutive 0-bits.
CRC-8: 8-bit checksum of the payload.

Receiving nodes that are configured to decode steganographic messages listen for the frame start pattern in the LSB stream. Once synchronized, they extract the frame length, payload, CRC, and verify integrity.

12.4 Encryption Layer

The hidden message is encrypted before steganographic encoding. The encryption uses a simple XOR cipher with a pre-shared key. While XOR is not cryptographically strong against a determined attacker with many captured messages, the steganographic layer provides the primary security (the attacker cannot even find the ciphertext), and the XOR layer provides defense in depth.

For higher security, the key can be derived from a passphrase using a key derivation function implemented in vanilla JavaScript. PBKDF2 can be implemented using the Web Crypto API (which is a browser built-in, not an external library) with SHA-256.

12.5 Plausible Deniability

If a device running the steganographic module is examined, the code must not obviously reveal its steganographic capability. The module should be designed so that:

The steganographic encoder and decoder are implemented as generic "data smoothing" functions that could plausibly be noise filters.

The configuration for steganographic mode is stored encrypted in IndexedDB under a generic key name.

The UI does not show any visible steganographic messaging interface unless the user enters a specific access code.

The deletion function (Principle 10) erases all steganographic configuration and message history instantly.


SECTION 13: ENVIRONMENTAL MESH (MODULE 11)

13.1 Distributed Weather Station

When many nodes across an area report temperature, pressure, and humidity, the aggregate data creates a high-resolution local weather map that no single weather station can provide.

The Environmental Mesh module collects temperature and barometric pressure readings from all nodes in the mesh (broadcast in CALM mode heartbeats) and computes spatial averages, gradients, and trends.

Temperature gradient: If nodes on the east side of the mesh consistently read 2 degrees warmer than nodes on the west side, this indicates a temperature front moving eastward.

Pressure gradient: Spatial pressure differences indicate wind direction (air flows from high to low pressure). Combined with the swarm coverage (200 nodes covers 2-3 square kilometers), this provides hyperlocal wind information valuable for wildfire response.

Precipitation detection: Rain increases ambient noise in the 500-2000 Hz range with a characteristic broadband spectrum. The audio analysis module can detect rain onset. If rain is detected by 30% or more of nodes in the mesh, the weather module reports active precipitation.

13.2 Air Quality Monitoring

Some Android devices have built-in gas sensors or particulate sensors. Even without dedicated hardware, smoke and pollution affect visibility and light levels, which the Ambient Light Sensor can detect as a reduction in light level inconsistent with time of day.

The air quality module maintains a baseline light level profile for each hour of the day. If the current light level is significantly below the expected baseline (more than 30% reduction) and this reduction is confirmed by multiple nodes, the module reports degraded air quality.

If dedicated air quality sensors are connected via BLE (some commercial particulate sensors support BLE), the module can ingest their readings and incorporate them into the swarm data.

13.3 Epidemic Early Warning

This is an ethically sensitive module and must be implemented with extreme care.

The principle: during an epidemic, affected populations change their movement patterns. People who are ill stop moving and cluster at home or in healthcare facilities. If BLE-equipped phones show a sudden change in population density patterns, specifically an unexpected increase in stationary phones in residential areas during hours when people would normally be active, this can be an early indicator of disease spread.

The module does NOT track individual people. It does NOT store location histories. It does NOT identify sick individuals. Instead, it monitors aggregate statistics: average node mobility (based on RSSI change rates across the mesh, without tracking which node is which), spatial density distribution (are nodes more clustered than usual?), and temporal activity patterns (are nodes less active during normally active hours?).

The ethical constraints are hard-coded and cannot be overridden by configuration:

No individual node tracking. The statistics are computed only in aggregate.
No storage of movement data beyond a 1-hour rolling window.
No transmission of any individual node data to Gateways. Only aggregate statistics are transmitted.
The module can be completely disabled with a single configuration toggle.
All data is deleted instantly by the deletion function.

The alert threshold is calibrated conservatively to avoid false alarms. The module reports an anomaly only when aggregate mobility drops by more than 40% from the seasonal baseline for more than 3 consecutive days. This is a very strong signal unlikely to be produced by normal variation.

13.4 Crowd Crush Prevention

This module addresses the catastrophic risk of crowd crush in dense gatherings. The 2022 Itaewon tragedy in Seoul demonstrated that crowd crushes can kill hundreds of people in minutes, and early warning could save many lives.

The detection principle: BLE devices in a crowd produce a measurable density signal. Each phone running the app can count the number of other BLE devices it can see and estimate the RSSI-based density. When the density exceeds safe thresholds, the system warns.

Crowd density estimation: Each node counts the number of unique BLE advertisements received per scan cycle (approximately every 5 seconds). This count, combined with the average RSSI (closer devices indicate denser packing), produces a local density estimate.

Safe crowd density: fewer than 4 people per square meter. The BLE count threshold for this depends on the mesh density but can be calibrated.

Dangerous density: 6-8 people per square meter. At this density, individuals begin losing the ability to move freely.

Critical density: above 8 people per square meter. At this density, crowd forces can exceed what a human body can withstand.

When the local density estimate enters the dangerous range, the module emits an alert. If multiple nodes in proximity confirm the density, the alert escalates.

The alert can trigger: a vibration warning on the user phone, an audible warning through the phone speaker, and a swarm-level alert relayed to Gateways for emergency services notification.

Crowd flow detection: By monitoring the RSSI change patterns of nodes in the crowd, the module can estimate crowd flow direction and velocity. If the flow velocity drops to zero while density remains high, this indicates a compression point, which is the immediate precursor to a crush event.

13.5 Dead Reckoning Navigation

In GPS-denied environments (indoor, underground, GPS-jammed areas), the mesh can provide approximate navigation using BLE RSSI relative positioning.

The principle: If a user knows the approximate positions of fixed BLE beacons in the mesh, they can estimate their own position based on the RSSI readings from nearby beacons. This is not precise (accuracy of 5-15 meters depending on beacon density) but is sufficient for directional guidance.

The dead reckoning module combines: BLE RSSI ranging from known-position nodes, smartphone compass heading, step counting from accelerometer (pedometer), and previous position estimates.

These inputs are fused using the Kalman Filter to produce a running position estimate. The estimate drifts over time (accumulated error from imprecise measurements), but periodic corrections from BLE beacons at known positions limit the drift.

The UI shows a simple arrow pointing in the estimated direction toward a configured destination, along with an estimated distance. This is sufficient for navigation in emergency scenarios (evacuating a building, navigating through rubble, finding an exit in smoke-filled conditions).


SECTION 14: SAFETY FENCE AND CROWD MONITOR (MODULE 12)

14.1 Invisible Fence for Children

The child safety fence is architecturally identical to the livestock virtual fence (Module 05, Section 7.3) but with adaptations for human use.

Boundary beacons are placed at the perimeter of a safe area (playground, campsite, refugee camp section). The child carries a BLE-equipped device (could be a cheap BLE tag, a smartwatch, or a phone).

When the child device moves beyond BLE range of all boundary beacons for more than 30 seconds, an alert is generated.

The alert is sent to a configured parent or guardian device through the mesh. The alert includes: the child device identifier, the last seen boundary beacon (indicating which direction the child left from), and the approximate time of departure.

Ethical constraints for child safety:

No location tracking history is stored beyond the immediate alert data.
No audio or video surveillance capability.
The fence can only detect presence or absence within a boundary, not track movement within the boundary.
All data is deleted instantly by the deletion function.
The child must be informed that they are carrying a device that will alert their guardian if they leave the area. Covert tracking of children is not supported by this module.

14.2 Elderly or Vulnerable Person Monitoring

The same invisible fence technology can be applied to monitor elderly persons with dementia or other conditions that may cause them to wander. The implementation is identical to the child fence with age-appropriate alert messaging.

The key addition for elderly monitoring is a fall detection algorithm. Using the accelerometer, the module detects the characteristic signature of a fall: a sudden high-g impact (more than 3g) followed by a period of stillness (no significant movement for more than 10 seconds). If a fall is detected and the person does not resume movement within 60 seconds, an automatic SOS alert is generated.

SECTION 16: ENHANCED IDEAS FROM IDEA-AMPLIFIER-ULTIMATE

The following ideas were generated through first-principles decomposition and cross-domain analogy. Each idea extends the base swarm functionality in a novel direction that does not exist in current commercial or research BLE mesh implementations.

16.1 Fungal Network Energy Sharing Protocol

Cross-domain analogy: Mycorrhizal networks in forests share nutrients between trees. Strong trees feed weak trees through underground fungal connections. The forest survives because no individual tree is allowed to die from resource starvation while neighbors have surplus.

Application: BLE mesh nodes have varying battery levels. A node with 80% battery can afford to relay more messages. A node with 10% battery should conserve energy. But the current system only reduces relay probability for low-battery nodes, which reduces their usefulness to the network.

The energy sharing protocol takes a different approach. High-battery nodes actively take over relay duties from low-battery neighbors. When a high-battery node detects a neighbor with battery below 15%, it broadcasts a "relay volunteer" signal. The low-battery node, upon receiving this signal, reduces its relay probability to near zero and extends its heartbeat interval to minimum duty cycle. The high-battery node compensates by increasing its own relay probability and reducing its jitter delay.

This redistributes the energy burden across the network, extending the operational lifetime of the weakest nodes. In biological terms, the mesh exhibits "resource redistribution through local cooperation" identical to fungal networks.

Implementation: approximately 40 lines of JavaScript added to the Core Mesh Engine. The decision logic is: if any neighbor battery below 15% and own battery above 40%, emit relay volunteer signal and increase own relay probability by the ratio of (own battery minus threshold) divided by own battery.

16.2 Acoustic Source Triangulation for Threat Localization

Already detailed in Module 09 (Section 11). This capability was derived from the insight that gunshot detection is easy (sudden loud sound), but gunshot localization (where did it come from?) is the information that saves lives. Three nodes with known positions and sub-millisecond timing can localize a sound source to within 20-50 meters.

16.3 Swarm Weather Station

Already detailed in Module 11, Section 13.1. The key insight is that 200 smartphones reporting temperature and pressure create a weather sensor network that rivals professional meteorological infrastructure in spatial resolution, at zero cost.

16.4 Epidemic Early Warning Through Mobility Analysis

Already detailed in Module 11, Section 13.3. The critical ethical constraint is that this module monitors aggregate mobility statistics, never individual movements. The module is designed so that even if the source code is examined, there is no capability to track individual people.

16.5 Invisible Safety Fence for Children

Already detailed in Module 12, Section 14.1. The adaptation from livestock fencing to child safety required adding: fall detection, mandatory informed consent (the child knows about the device), and strict data minimization (no location history).

16.6 Steganographic Communication for Activists

Already detailed in Module 10 (Section 12). The 4-bit-per-heartbeat encoding achieves communication that is undetectable even by sophisticated BLE traffic analysis, at the cost of very low data rate. This trade-off is intentional: for activists in surveillance environments, undetectability is more valuable than speed.

16.7 Crowd Crush Prevention System

Already detailed in Module 11, Section 13.4. This capability was inspired by the 2022 Itaewon tragedy analysis which showed that crowd density reached lethal levels over a period of 15-20 minutes before the crush occurred. A BLE density monitoring system could have detected the dangerous density buildup and issued warnings with sufficient lead time for intervention.

16.9 Soil Moisture and Flood Prediction Network (New Enhanced Idea)

Cross-domain analogy: Earthworms sense soil saturation and surface en masse before floods. The swarm can replicate this by monitoring proxy indicators of soil moisture.

Application: Many Android devices report ambient humidity through the Generic Sensor API. While this measures air humidity, not soil moisture, there is a well-established correlation: as soil approaches saturation, local humidity rises, especially near the ground.

If BLE sensor nodes are deployed at ground level (livestock tags, fixed environmental sensors), their humidity readings can serve as a distributed soil moisture monitoring network. When a critical fraction of ground-level nodes report humidity above 95% for more than 6 hours, the system flags potential flood risk.

Combined with barometric pressure data (falling pressure plus high humidity strongly predicts heavy rainfall), the network can issue flood warnings hours before water levels rise.

This is particularly valuable in refugee camps and informal settlements located in flood-prone areas, where no official flood warning infrastructure exists.

Implementation: approximately 30 lines of JavaScript. Uses existing humidity and pressure data from heartbeat payloads. Computes a flood risk score based on the composite of: percentage of nodes reporting high humidity, rate of pressure decrease, and duration of high-humidity conditions.

16.10 Silent Barter and Resource Exchange Protocol (New Enhanced Idea)

Cross-domain analogy: Hawala, the informal value transfer system used across South Asia and the Middle East, operates without any central ledger. Trust between agents, not institutional infrastructure, enables transactions.

Application: In refugee camps or post-disaster environments, people need to exchange resources (food, medicine, tools, information) but may lack any communication infrastructure beyond the BLE mesh.

The resource exchange protocol allows nodes to broadcast anonymized resource offers and requests. A node can advertise "have: water purification tablets, need: bandages" using a compact code system (each common resource is assigned a 1-byte code). Other nodes can respond with matching offers.

The exchange is coordinated through a BLE-based rendezvous protocol. Once two nodes agree on an exchange, they switch to a direct BLE connection for coordination (meeting location, quantity, timing).

No personal identity is attached to offers or requests. No transaction history is stored beyond the immediate exchange. No central authority mediates or records transactions.

Ethical constraint: This module explicitly does not support currency or monetary exchange, to prevent exploitation. It supports only barter of physical goods and information.

Implementation: approximately 60 lines of JavaScript. Uses a 1-byte resource code table (256 possible resources), a 1-byte quantity field, and a 1-bit offer/request flag. Each advertisement carries one offer or request alongside the normal heartbeat data.

16.11 Acoustic Biodiversity Monitoring for Deforestation Detection (New Enhanced Idea)

Cross-domain analogy: Canaries in coal mines. When birds stop singing, the ecosystem is dying.

Application: Forest ecosystems produce a characteristic soundscape dominated by bird calls, insect sounds, and wind through canopy. Deforestation and environmental degradation cause measurable changes in this acoustic biodiversity index.

The system maintains a rolling baseline of the acoustic diversity (number of distinct frequency peaks in the 500-8000 Hz range, measured daily at dawn when bird activity peaks). A sustained reduction in acoustic diversity (fewer distinct frequency peaks over weeks) indicates ecosystem degradation.

This monitoring requires no specialized hardware. Any BLE node with a microphone (smartphone, wildlife camera with BLE) can contribute.

The alert is not urgent (it measures slow changes over weeks to months) but provides valuable environmental monitoring data for conservation organizations.

Implementation: approximately 40 lines of JavaScript. Daily dawn recording (5 minutes at configured sunrise time), FFT analysis counting peaks above noise floor, comparison to 30-day rolling baseline.

SECTION 21: ERROR SCENARIOS AND EDGE CASES

21.1 What happens when a node runs out of battery mid-operation?

The power management state machine monitors battery level continuously. At 20% battery, the node reduces activity (longer heartbeat intervals, lower relay probability). At 10%, the node enters minimal mode (heartbeat every 60 seconds, relay only PANIC messages). At 5%, the node emits a final "going offline" heartbeat and shuts down gracefully, saving all state to IndexedDB for resumption when power is restored.

If the battery dies before graceful shutdown (sudden power loss), the node simply disappears from the mesh. Neighbors detect the absence through missing heartbeats and update the node record to "offline" after 3 consecutive missed expected heartbeats.

21.2 What happens when the mesh splits into disconnected fragments?

The mesh can fragment when Relay nodes fail or when the geographical gap between node clusters exceeds BLE range. Each fragment continues to operate independently. Alerts are contained within the fragment. SOS messages are queued for relay.

When fragments reconnect (a new node bridges the gap, or a mobile node moves between fragments), queued messages are exchanged during the reconnection period. The reconciliation protocol exchanges the most recent N messages (configurable, default 50) to synchronize state across fragments.

21.3 What happens if an adversary jams BLE frequencies?

BLE operates in the 2.4 GHz ISM band and uses frequency hopping across 40 channels. Jamming all 40 channels simultaneously requires significant power and equipment. However, it is possible.

If BLE communication fails completely, the node falls back to local-only operation. It continues monitoring sensors and logging data to IndexedDB. When jamming ceases, the node resumes mesh participation and transmits accumulated alerts.

The system does not attempt to defeat jamming. Countering electronic warfare is beyond the scope of a smartphone BLE application. The system acknowledges this limitation honestly.

21.4 What happens if an adversary deploys fake nodes?

Fake nodes (Sybil attack) are mitigated by the Friendship Trust system. New nodes start at trust 0.5 and must earn trust through consistent behavior over time. A sudden influx of new nodes with suspicious behavior (all appearing simultaneously, all broadcasting similar data, all located in the same position) triggers a trust alert.

The system cannot completely prevent Sybil attacks if the adversary is patient and sophisticated. The trust system raises the cost and time required for an attack but does not make it impossible. This limitation is acknowledged in the code comments and documentation.

21.5 What happens when many nodes in the mesh use different software versions?

The protocol version byte in the advertisement header (byte 2) enables version negotiation. A node receiving an advertisement with an unknown protocol version ignores it silently rather than crashing. A node receiving an advertisement with an older protocol version processes it according to the older version rules (backward compatibility).

Version upgrades propagate through the mesh informally. When a user upgrades their software, their node begins advertising with the new version number. Other nodes do not need to upgrade simultaneously.

21.6 What about false alarms?

False alarms are inevitable in any monitoring system. The design philosophy is to minimize false alarms through multi-sensor fusion and swarm consensus while acknowledging that some false alarms will occur.

The system provides a false alarm feedback mechanism in the UI. When the user identifies a false alarm, they tap a "false alarm" button. This feedback adjusts the local Bayesian priors downward for the relevant alert type, making the system less sensitive to that type of event in the future.

False alarm rates are logged and displayed in the dashboard statistics, enabling the user to assess the reliability of the system and adjust thresholds accordingly.

SECTION 25: GLOSSARY

BLE: Bluetooth Low Energy. A wireless communication standard designed for short-range, low-power operation.

Bloom Filter: A probabilistic data structure that can test whether an element is a member of a set. False positives are possible, but false negatives are not.

CALM: The default low-power operating mode of the swarm.

ALERT: The elevated monitoring mode activated when anomalies are detected.

PANIC: The maximum urgency mode for confirmed life-threatening situations.

Chinbun: Friendship in Japanese. The name of the trust protocol reflects the idea that trust is earned through sustained positive interactions, like friendship.

FFT: Fast Fourier Transform. An algorithm that converts a time-domain signal into its frequency components.

GATT: Generic Attribute Profile. A BLE communication protocol for exchanging data between connected devices.

Gateway: A node that bridges the BLE mesh to external networks (cellular, WiFi, LoRa).

Heartbeat: A periodic BLE advertisement that serves as the fundamental communication unit of the swarm.

IIFE: Immediately Invoked Function Expression. A JavaScript pattern that creates a private scope to prevent global namespace pollution.

IndexedDB: A browser-based NoSQL database for storing structured data on the client side.

Kalman Filter: A mathematical algorithm that estimates the state of a system from noisy observations.

LSB: Least Significant Bit. The bit in a binary number that represents the smallest value.

Managed Flooding: A message propagation strategy where each node probabilistically decides whether to relay a message, balancing coverage against redundancy.

RSSI: Received Signal Strength Indicator. A measurement of the power level of a received BLE signal, used as a proxy for distance.

Sentinel: A node that carries sensors and generates data for the swarm.

Steganography: The practice of hiding a message within another message or object so that the existence of the hidden message is not apparent.

SwarmBus: The global event bus that enables inter-module communication.

Sybil Attack: An attack where an adversary creates many fake identities to subvert a reputation or voting system.

TDOA: Time Difference of Arrival. A method for locating a signal source by comparing the arrival times at multiple receivers.

TTL: Time to Live. A counter that decrements at each relay hop, preventing messages from circulating indefinitely.

Web Bluetooth API: A browser API that allows web pages to communicate with BLE devices. Available on Chrome for Android and desktop. Not available on iOS.

SECTION 28: KNOWN LIMITATIONS

This skill and the systems it generates have the following known limitations that must be communicated honestly to all users and stakeholders.

28.1 iOS Exclusion

Web Bluetooth is not supported on iOS. This excludes all iPhones and iPads from participating as active nodes. iOS devices can serve as passive presence indicators (their BLE advertisements are visible to Android nodes) but cannot run the swarm software. This is a platform limitation imposed by Apple, not a design choice.

28.2 Background Operation

Web Bluetooth scanning stops when the browser is in the background or the screen is off. The Screen Wake Lock API keeps the screen on, but this increases power consumption. For true background operation, the web application must be wrapped in a native container (Capacitor, TWA) which adds deployment complexity.

28.3 Range Limitations

BLE typical range is 50-100 meters in open environments and 10-30 meters indoors. The mesh extends range through multi-hop relay, but each hop adds latency and reduces reliability. Very long mesh chains (more than 15 hops) may experience significant message loss.

28.4 No Guaranteed Delivery

The mesh uses probabilistic relay, not guaranteed delivery. Messages may be lost due to: simultaneous transmission collisions, all relay candidates deciding not to relay (probability-based), or network partition. SOS messages mitigate this with probability 1.0 relay and high TTL, but even SOS delivery is not guaranteed in a severely fragmented or jammed network.

28.5 Acoustic Analysis Accuracy

Audio-based detection (wildfire crackling, gunshots, livestock distress, locusts) operates on low-quality smartphone microphones in noisy outdoor environments. False positive rates will be higher than laboratory conditions. Multi-sensor fusion and swarm consensus significantly reduce false positives but cannot eliminate them entirely.

28.6 Power Consumption

While CALM mode is very power-efficient, ALERT and PANIC modes consume significantly more power. A node in PANIC mode may drain its battery in 6-12 hours. Power management mitigates this but cannot eliminate the fundamental trade-off between responsiveness and battery life.

28.7 Security Limitations

The system is designed for community safety, not military-grade security. The steganographic communication provides concealment against casual observation but may not withstand forensic analysis by sophisticated state-level adversaries. The trust system raises the cost of Sybil attacks but does not make them impossible. The BLE protocol itself is not encrypted at the application layer (heartbeat data is broadcast in the clear), though the steganographic module encrypts its hidden payload.

These limitations are documented honestly because Principle 2 (Ensure transparency) requires acknowledging what the system cannot do, not just what it can do.
