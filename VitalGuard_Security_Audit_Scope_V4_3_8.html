1. Purpose and Audit Goal

VitalGuard AI V4.3.8-END is a single-file, offline-first humanitarian technology demonstration intended for low-connectivity, outage-prone, and off-grid contexts. The application makes strong user-facing claims: no cloud dependency, no telemetry, local-only storage, user-controlled deletion, Bluetooth proximity monitoring, encrypted backup, copy-only sharing, redacted diagnostics, and disabled worker/service-worker behavior by default.

The audit goal is to produce a precise security verdict for the frozen V4.3.8 artifact: which claims are supported, which are supported only under defined browser/deployment contexts, and which require remediation or clearer user guidance.

note. the security posture now centers on nonce CSP, script-src-attr 'none', strict data-vg-on* dispatch, Trusted Types / sink scrubbing, runtime network API kill-switches, encrypted-only default backup, bounded import, redacted diagnostics, and service worker disabled by default.

4. Core Audit Questions

  1. Does the artifact generate any application-initiated network egress during normal launch, tag registration, BLE monitoring, SOS, geolocation save, voice tools, encrypted backup/export/import, diagnostics, PWA install, or offline restart?
  2. Does the CSP and runtime network guard actually prevent fetch/XHR/WebSocket/EventSource/sendBeacon/form submission, and are browser/platform/deployment requests correctly separated from application-authored traffic?
  3. Are all HTML/event/action flows compatible with script-src-attr 'none' and strict data-vg-on* dispatch without enabling eval/new Function or unsafe dynamic code?
  4. Can attacker-controlled strings from BLE advertisements, backups, rescue packs, QR payloads, emergency fields, diagnostics, or i18n/help/legal content reach the DOM unsafely?
  5. Are encrypted backups, rescue packs, and import/merge/replace flows robust against malformed input, oversized data, schema abuse, voice-blob abuse, state poisoning, and partial restore failures?
  6. Are IndexedDB, localStorage fallback keys, blobs, Cache Storage, reset/factory reset, auto-wipe-on-close, and browser residuals accurately described as guaranteed versus best effort?
  7. Are Bluetooth, geolocation, microphone, motion sensors, notification, clipboard, wake lock, and speech synthesis only invoked through clear user action and safe permission flows?
  8. Do on-device ML components and persisted model state resist poisoning, leakage, resource exhaustion, or alert suppression within realistic browser constraints?
  9. Are open-source/FOSS and public-interest controls represented accurately as notice/integrity/auditability mechanisms rather than absolute abuse-prevention guarantees?

5. Proposed Work Packages

WP0 — Artifact freeze and claim alignment

Record exact filename, SHA-256, byte size, line count, browser/runtime assumptions, and the intended meaning of “zero egress” for local, localhost, HTTPS, and hosted deployments.

WP1 — Zero-egress and network runtime verification

Search source for network primitives; instrument launch, BLE scan, SOS, geolocation, voice, backup/import, diagnostics, install, and offline restart; verify RuntimeNetworkGuard and CSP behavior.

WP2 — CSP, Trusted Types, action dispatch, and DOM sink security

Review nonce CSP, script-src-attr none, Trusted Types policy, mutation scrubber, action allowlist, blocked URL protocols, document.write block, and injected HTML compatibility.

WP3 — Sensitive browser API surface

Map Web Bluetooth, geolocation, microphone/getUserMedia, notifications, clipboard, wake lock, motion events, and speech synthesis; verify clear user action, denial handling, and local-only behavior.

WP4 — Local persistence, reset, factory reset, and auto-wipe

Review IndexedDB, localStorage fallback, blob storage, full wipe, reset center, auto-wipe-on-pagehide, cache deletion, service worker unregister calls, and residual browser artifacts.

WP5 — BLE privacy, matching safety, and radio-side detectability

Review requestLEScan filters/fallback, registration scan 30-second consent flow, matching thresholds, collision handling, crowded BLE behavior, scan restart jitter, and nearby adversary detectability.

WP6 — Encrypted backup, hardened import, and Rescue Pack

Review CryptoBox, encrypted-only default export, import size/depth/schema limits, voice clip limits, rescue pack expiry/schema, copy-only sharing, and state poisoning resistance.

WP7 — DOM injection and UI spoofing

Inject hostile payloads through pet names, BLE names, emergency profile fields, backups, rescue packs, diagnostics, i18n strings, QR payloads, and coach actions.

WP8 — On-device ML and model-state risk

Review KalmanRSSI, QLearningLite, KNNLite, IsolationForestLite, RLSCalibrator, BehavioralFingerprint, and import/reset behavior for poisoning, leakage, and resource exhaustion.

WP9 — Diagnostics, observability, and error safety

Validate redacted diagnostics, self-test, ErrorShield behavior, crash-safe logs, clipboard/download exports, and claim language around local-only diagnostics.

WP10 — Open-source/FOSS notice and non-abuse claim language

Confirm Apache-2.0 / SPDX metadata, public-interest guidance, no field-of-use restrictions, manifest integrity diagnostics, and non-overstated user-facing claims.

7. Suggested Out-of-Scope Boundaries

  • General UX polish, branding, visual design, copy editing, or non-security product feedback unless it affects security claims.
  • BLE range accuracy as a product-quality metric, animal-safety validation, or certification of life-critical reliability.
  • Backend, server, build pipeline, repository, or infrastructure review outside the frozen single-file artifact, except where hosted runtime behavior affects zero-egress, CSP, or browser security claims.
  • Future revisions, prior V4.2.x/V4.3.7 revisions, or delta comparison except where the V4.3.7 documents are used to confirm that this V4.3.8 scope was updated correctly.
  • Commercial, fundraising, governance, or organizational review beyond technical claims that affect user safety, security, privacy, or FOSS compatibility.

8. Reporting Requirements

For each material finding, the report should include:

  • Affected artifact name, SHA-256, execution context, browser/OS, and exact reproduction steps.
  • Classification as source-level, runtime-level, deployment-context-dependent, or user-action-dependent.
  • Security claim affected, realistic impact, exploitability, and recommended severity.
  • Evidence, including static source references, DevTools/network captures, CSP violation logs, storage snapshots, permission prompts, adversarial payloads, or import files.
  • Concrete remediation guidance and retest steps with expected passing behavior.
  • A final claim verdict table using: Supported, Supported with caveats, Partially supported, Not supported, or Not applicable.
